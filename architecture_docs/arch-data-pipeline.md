## Overview

The data pipeline collects BTC/USD OHLCV candles from multiple exchanges, merges them into a unified price series, and computes relative volume profiles (VP) that serve as the primary feature for the ML model.

## Data Sources

- **Exchanges**: Bitstamp and Coinbase (via ccxt library)
- **Symbol**: BTC/USD
- **Timeframe**: 4h candles
- **History start**: 2016-01-01
- **Merge strategy**: outer join on timestamp; prices averaged across exchanges (non-zero only), volume summed

### Why multiple exchanges?
Single-exchange data has gaps and volume skew. Merging Bitstamp + Coinbase gives more complete coverage and a more representative volume signal. Additional exchanges can be added to the `EXCHANGES` config list.

## Relative Volume Profile (VP)

The core feature: a 50-bin histogram of where volume traded relative to the current price, over a lookback window.

### How it works
For each 4h candle at time `t`:
1. Take the past 180 days (1,080 bars) of candles
2. For each historical candle `j`, compute `x_j = log(close_j / close_t)` — the log-return distance from current price
3. Bin these distances into 50 equally-spaced bins on the log axis, spanning ±25% from current price
4. Weight each bin by the volume at that bar
5. Normalize so the histogram sums to 1.0

### What the VP tells the model
- **Peaks** in the VP = price levels where heavy volume traded (support/resistance zones)
- **Symmetry** of the VP = whether more volume is above or below current price
- **Spread** = how dispersed recent trading activity is around current price

### Key parameters
| Parameter | Value | Meaning |
|-----------|-------|---------|
| `LOOKBACK_DAYS` | 180 | History window for VP computation |
| `REL_SPAN_PCT` | 0.25 | ±25% price range around current close |
| `REL_BIN_COUNT` | 50 | Number of histogram bins |
| `REL_NORMALIZE` | True | Histogram sums to 1.0 |

### Column naming
VP columns: `vp_rel_00` through `vp_rel_49`
- Bin 00 = lowest price bucket (−25% from current)
- Bin 25 = center (current price)
- Bin 49 = highest price bucket (+25% from current)

## Data Flow

```
Bitstamp API ──┐
               ├── merge (outer join) ── compute VP ── feature CSV
Coinbase API ──┘
```

## Output Format

CSV file (`BTC_4h_RELVP.csv`) with columns:
- `ts` — Unix timestamp in milliseconds
- `date` — ISO 8601 datetime (UTC)
- `open`, `high`, `low`, `close` — merged OHLC prices
- `volume_4h` — merged volume (sum across exchanges)
- `vp_rel_00` ... `vp_rel_49` — relative VP histogram (50 bins)

## Edge Cases & Gotchas

- **First 1,080 rows are discarded** — VP requires a full lookback window, so the first 180 days of data have no VP values
- **Zero-volume bins** — some VP bins will be zero if no trading happened in that price range; this is expected, not an error

## Alternate VP file: `BTC_1h_RELVP_30d.csv` (v10 experiment)

A second VP file with `LOOKBACK_DAYS=30` is generated by
`python3 -m src.data.compute_vp_30d`. Same bin count, span, and
normalization — only the history window shrinks from 180d → 30d.

Rationale: the default 180d VP blends old-regime volume into the
histogram, diluting the recent support/resistance signal that the user
actually trades on. The 30d variant sharpens recency to match the
chart-reader view. Paired with the v10 model's 90-day temporal window,
the net history-per-forward-pass is 120d (vs 210d on v6-prime) but
training gains ~2,100 rows because the warmup shrinks.

Both CSVs coexist; legacy evals (`eval_v6_prime`, `eval_v9*`) still
read the 180d file. Only `eval_v10` uses the 30d file.

## Absolute-range VP file: `BTC_15m_ABSVP_30d.csv` (v11 experiment)

A third, structurally different VP file generated by
`python3 -m src.data.compute_absvp_15m_30d`. This one is **not a
variant** of the relative-VP pipeline — it's a separate representation
with its own semantics.

Written at **15m resolution** (357,705 rows, 2016-01-31 → ongoing) with
absolute-range (visible-range) VP bins:

- **VP axis**: absolute price, linear. For each bar, 50 bins are placed
  linearly between the trailing 30-day **wick** low and wick high
  (using `low` / `high`, not `close`). Volume-weighted histogram of
  in-window closes, normalized to sum 1. Column names: `vp_abs_00` …
  `vp_abs_49`. Bin 0 is always the lowest-price bucket; the current
  close can sit anywhere in [0, 49] per row.
- **Self channel**: hard one-hot over which bin contains `close_t`.
  Column names: `self_00` … `self_49`. Exactly one bin is 1.0, rest 0.0.
  Gives downstream models a positional anchor that was implicit (bin 25
  = close) under relative VP.
- **Scalars**:
  - `price_pos` = `(close − window_lo) / (window_hi − window_lo)` ∈ [0, 1]
  - `range_pct` = `(window_hi − window_lo) / close`
- **Debug columns**: `window_lo`, `window_hi` (the wick range used for
  binning, per row), kept for chart sanity-checking.

Rationale: this mirrors how Kraken's VRVP is actually read — the 50
bins span the real 30-day `[low, high]` range, not a fixed ±25% window
centered on current price. Peaks in this histogram are at absolute
price levels trackable across rows, and the self-channel lets the
spatial transformer reason about "N bins from me to the nearest peak"
directly. See `experiments/LABEL_REDESIGN.md` for why this pipeline
was built and why the first model on it (v11) was rejected despite the
representation change.

Only `eval_v11` reads this file. Legacy evals are unaffected.

### Data hygiene — the v11 label leak cautionary tale

Any model that uses `window_hi` / `window_lo` as features **must not**
also use them to define labels. The v11 range-derived labels
(`TP_pct = (window_hi − close)/close × 0.8`, clipped) introduced a
structural asymmetry/label coupling that was invisible until post-hoc
filter analysis exposed it. On 170k samples, pos_rate moved
monotonically from 88.5% (low asymmetry) to 18.6% (high asymmetry),
making labels near-deterministic from range geometry alone. A free
classifier using only `asym` scored ~80% accuracy. See
`experiments/LABEL_REDESIGN.md` and `EVAL_AUDIT.md` §9 for the full
explanation.

**Rule going forward**: labels computed in any training script must
not be a function of features the model sees at the same index.
Volatility-scaled triple-barrier labels are the standard way to avoid
this and are the next planned experiment.
- **Exchange downtime** — if one exchange has no data for a timestamp, the merge uses the other exchange's price (not averaged). Volume from the missing exchange is 0 for that bar
- **Partial candles** — the last candle in a live data pull may be incomplete (candle hasn't closed yet). The scraper doesn't handle this; the real-time data feed (Phase 4) must
